# ファイル構成・肥大化対策ルール

## 目的

コードベースの成長に伴うファイル肥大化や構造劣化を防ぎ、保守性・可読性・発見性を維持することを目的とする。

本ルールは言語やアーキテクチャ（オニオンアーキテクチャ、クリーンアーキテクチャ、ヘキサゴナルアーキテクチャなど）に依存せず適用する。

---

# 基本原則

## 1. 共通化より局所化を優先する

コードは可能な限り、その責務を持つ機能やファイルの近くに配置する。

再利用されたという理由だけで安易に共通フォルダへ移動してはならない。

共通化は「本当に共通概念である」と判断できた場合のみ行う。

---

## 2. 肥大化したら分割する

ファイルが大きくなり、

* 理解しにくい
* 修正しにくい
* レビューしにくい

状態になった場合は責務ごとに分割する。

ただし機能単位のまとまりは維持すること。

---

## 3. 所有者を中心に構造化する

分割したコードが特定ファイルからしか利用されない場合、そのファイル専用のフォルダにまとめる。

例：

```text
user_page/
    user_page.ext
    header.ext
    list_view.ext
    action_panel.ext
```

補助ファイルは親ファイルの責務に属するものとして扱う。

---

# フォルダ化ルール

## 推奨構成

ファイルが肥大化して複数ファイルへ分割する場合は、親ファイル専用フォルダを作成する。

```text
user_page/
    user_page.ext
    user_header.ext
    user_list_view.ext
    user_action_panel.ext
```

---

## 非推奨構成

```text
user_page.ext
user_page/
    user_header.ext
    user_list_view.ext
```

理由：

* IDE上で親子関係が分かりにくい
* ファイルとフォルダが離れて表示される
* 関連ファイルの発見性が低下する

---

## 判断基準

以下の状態になったらフォルダ化を検討する。

* 分割ファイルが2個以上になった
* 今後さらに増加する可能性が高い
* ひとつの責務群として管理したい

---

# 昇格ルール

コードの配置は使用範囲によって決定する。

---

## 第1段階：Private（専用）

利用箇所が1つのみ。

配置場所：

```text
feature/
    target_file/
```

または同等のローカルフォルダ。

---

## 第2段階：Feature Shared（機能共有）

同一機能内の複数箇所から利用される。

配置場所：

```text
feature/
    common/
```

例：

```text
trade/
    common/
        price_formatter.ext
        trade_validator.ext
```

---

## 第3段階：Global Shared（全体共有）

複数機能から利用される。

配置場所：

```text
shared/
```

例：

```text
shared/
    validation/
    formatting/
    ui/
    domain/
```

---

# 共通フォルダの制約

共通フォルダを雑多な置き場にしてはならない。

禁止例：

```text
common/
    utils.ext
    helper.ext
    misc.ext
```

推奨例：

```text
shared/
    validation/
    formatting/
    ui/
    domain/
    infrastructure/
```

責務ごとに分類すること。

---

# 移行ルール

コードを移動する際、

* 一時的なラッパー
* export
* re-export

の利用は許可する。

ただし移行期間のみとする。

恒久的な互換レイヤーを作ってはならない。

削除予定コメントや Deprecated 指定を付与すること。

---

# 依存関係ルール

コードは使用範囲に応じて上位へ昇格する。

```text
Private
    ↓
Feature Shared
    ↓
Global Shared
```

逆方向への移動は原則禁止とする。

---

# AIの判断手順

コードを新規作成または修正する際は以下の順で判断する。

1. このコードの所有者は誰か
2. 利用箇所は何箇所か
3. 最も狭いスコープへ配置できないか
4. 共通化する必然性はあるか
5. フォルダ化した方が発見しやすくならないか
6. 将来的な保守性は向上するか

---

# デフォルト判断

判断に迷った場合は以下を優先する。

* 共通化しない
* 近くに置く
* 昇格を後回しにする
* 発見しやすさを優先する
* 保守性を優先する

短期的な便利さよりも長期的な管理容易性を重視すること。
