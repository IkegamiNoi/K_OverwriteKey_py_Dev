# タスク3 指示書

**題名:** 分離JSON保存の実装（ランタイム → keymap_set / keymaps / trigger_sets / hotkey_presets）

## 目的

タスク2で実装した**分離JSON読込**に対応して、
現在のランタイムデータ `self.data` から **分離JSON形式で保存できるようにする**。

このタスクでは、保存時に以下を出力する。

* `config/config.json`
* `config/user/keymap_sets/default.json`
* `config/user/keymaps/*.json`
* `config/user/trigger_sets/default.json`
* `config/user/hotkey_presets/default.json`

---

## このタスクの位置づけ

* タスク2で分離JSON読込は動作確認済み
* ただし保存はまだ従来の単一JSON中心
* このタスクでは **保存の主系を分離JSONへ移す**
* 旧単一JSON保存は、必要なら互換用として残してもよいが、主目的は分離保存

---

## 絶対条件

* フック処理を変更しない
* UIレイアウトを変更しない
* 既存ランタイム構造は維持する
* 最小差分で実装する
* 保存後、**そのまま再起動して読める分離JSON** を出力する
* 既存機能を壊さない
* JSON構造の互換を不用意に崩さない

---

# 設計の前提

## 1. 起動入口

`config/config.json` は **起動入口 + UI共通設定** とする。

このタスクで保存する内容の基本形:

```json
{
  "keymap_set_path": "user/keymap_sets/default.json",
  "ui_font_delta_pt": 0,
  "last_used_directory": ""
}
```

補足:

* `prompt_if_missing` 等、既存で起動設定として持っている値は必要に応じて保持してよい
* ただし運用本体は入れない

---

## 2. keymap_set.json は構成セットの設定本体

保存先:

```text
config/user/keymap_sets/default.json
```

基本形:

```json
{
  "trigger_set_path": "user/trigger_sets/default.json",
  "hotkey_presets_path": "user/hotkey_presets/default.json",
  "active_keymap_path": "user/keymaps/default.json",
  "keymaps": [
    {
      "path": "user/keymaps/default.json",
      "switch_key": "f6"
    }
  ],
  "hook_stop_key": "f12",
  "hook_toggle_key": "",
  "keyboard_layout": "us_tkl",
  "keyboard_show_physical_key_labels": false,
  "debug_jis_special_key_events": false,
  "external_keyboard_layouts": []
}
```

---

## 3. keymap は個別ファイル保存

保存先例:

```text
config/user/keymaps/default.json
config/user/keymaps/default_2.json
```

形式:

```json
{
  "label": "Default",
  "mappings": {}
}
```

補足:

* `id` は保存必須ではない
* 参照は path ベース
* ランタイムで使っている `id` は保存時に path へ戻す

---

## 4. trigger_set / hotkey_presets は今は単数保存

保存先:

```text
config/user/trigger_sets/default.json
config/user/hotkey_presets/default.json
```

形式:

```json
{
  "triggers": [...]
}
```

```json
{
  "hotkey_presets": [...]
}
```

将来複数化の余地は残してよいが、このタスクでは単数前提でよい。

---

# このタスクでやること

## 1. ランタイムから分離保存用データを組み立てる

`self.data` から以下を分離保存形式へ変換する。

* 起動設定 (`config/config.json`)
* keymap_set
* keymaps 個別
* trigger_set
* hotkey_presets

---

## 2. 保存時の主系を分離JSONへする

`save()` / `save_as()` などから保存する際、
主系として分離JSON一式を書き出すようにする。

---

## 3. 保存後に再読込可能な整合を保つ

重要:

* `config/config.json` の `keymap_set_path`
* `keymap_set.json` の `active_keymap_path`
* `keymap_set.json` の `keymaps[].path`
* `trigger_set_path`
* `hotkey_presets_path`

が、保存された実ファイルと一致していること。

---

# 実装方針

## 1. 保存責務は ConfigService を境界にする

分離保存の主処理は **ConfigService 側へ寄せる**。

想定責務:

* ランタイムから分離保存構造を構築
* 保存先ディレクトリ生成
* JSON書き出し
* 相対パス整形
* 既存起動設定との整合維持

App 側で分解保存ロジックを直接持たないこと。

---

## 2. configルート基準の相対パスで保存する

保存する path 文字列はすべて **configルート基準の相対パス** にする。

例:

* `user/keymap_sets/default.json`
* `user/keymaps/default.json`

絶対パスは保存しない。

---

## 3. keymap のファイル名決定ルールを用意する

ランタイムの `keymaps` を個別ファイルへ保存するため、
各 keymap に対応するファイル名を決める必要がある。

### 方針

* 既存 split 読込で path 由来の情報を保持できるなら、それを優先再利用してよい
* 無ければ以下の優先順で生成してよい

優先順:

1. keymap の `id`
2. `label`
3. `"keymap"`

ファイル名生成時は:

* `normalize_key_name()` 相当で安全化
* 重複時は `_2`, `_3` を付ける

例:

* `default.json`
* `default_2.json`

---

## 4. active keymap の path を決める

ランタイムでは `active_keymap_id` 互換で持っている可能性がある。
保存時には **対応する keymap ファイルの path** に戻して `active_keymap_path` に書く。

もし active が解決できなければ:

* 先頭 keymap を active にしてよい
* keymap が無ければ空文字でよい

---

## 5. keymap_switch_keys を keymaps[].switch_key へ戻す

ランタイムでは:

```python
data["keymap_switch_keys"] = {
    "f6": "default"
}
```

のような形になっている可能性がある。

保存時は、各 keymap エントリに対応する `switch_key` を復元して:

```json
{
  "path": "user/keymaps/default.json",
  "switch_key": "f6"
}
```

の形へ戻すこと。

---

## 6. external_keyboard_layouts は保存形式を維持する

`external_keyboard_layouts` は現在の実行時表現との整合を崩さず保存すること。
可能なら既存形式を維持する。

path を含む場合は必要に応じて config ルートまたは従来互換の相対化を保つこと。
既存動作を壊しそうなら最小差分優先。

---

## 7. ディレクトリが無ければ作成する

保存前に以下を作成してよい。

* `config/`
* `config/user/`
* `config/user/keymap_sets/`
* `config/user/keymaps/`
* `config/user/trigger_sets/`
* `config/user/hotkey_presets/`

---

## 旧 config/user/config.json の扱い

このタスクでは次のどちらでもよい。

### 許容案A

分離JSON保存を主系にしつつ、旧単一JSON保存も当面残す

### 許容案B

保存時は分離JSONのみ更新し、旧単一JSONは更新しない

ただし、どちらを選んだかは差分説明で明記すること。

推奨は **A**。
理由:

* 互換確認がしやすい
* 万一の切り戻しがしやすい

ただし、保存ロジックが複雑になりすぎるなら **B** でもよい。
最小差分優先。

# App 側の変更方針

App 側は既存の保存導線を極力維持し、
**ConfigService に分離保存を委譲する**形を優先する。

つまり App 側では:

* 保存先の入口パス確定
* dirty 制御
* ダイアログ表示

程度に留める。

分離保存の中身は ConfigService 側で行うこと。

# 動作確認項目

## 基本

1. 既存データを読み込んだ状態で保存すると、分離JSON一式が出力される
2. 出力後に再起動すると、分離読込で同じ状態が復元される
3. 停止キー、レイアウト、trigger、preset、keymap が維持される
4. keymap の `switch_key` が保存される
5. active keymap が保存・復元される

## keymap

6. keymap が1件でも保存できる
7. keymap が複数ある場合、ファイル名衝突を避けて保存できる
8. `keymaps[]` の順序が保存後も維持される

## フォールバック/互換

9. 旧単一JSON読込を壊していない
10. 必要に応じて旧単一JSON保存も維持できている

## 既存機能

11. UI表示が壊れていない
12. フック動作が壊れていない
13. direct switch が壊れていない

---

# このタスクでやらないこと

* 個別保存
* keymap_set のUI切替
* trigger_set / hotkey_presets の複数保存
* 旧 settings からの移行機能
* 過剰なリファクタ
* ConfigService 全面再設計

---

# 補足判断

## 1. 今は default 固定でよい

保存先の keymap_set / trigger_set / hotkey_presets は
今は `default.json` 固定でよい。

## 2. keymap path は順序安定を優先

将来 rename を考えすぎず、今は**再読込できること**と**順序維持**を優先する。

## 3. path情報をランタイムに一時保持してよい

保存時の対応付けのため、読込時に keymap ごとの source path を内部で保持する方法は許容。
ただし既存UIや互換を壊さないこと。
