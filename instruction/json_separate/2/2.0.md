# タスク2 指示書

**題名:** 分離JSON読込の実装（config.json → keymap_set.json → 各実体JSON）

## 目的

単一 `config/user/config.json` を直接読む現在の実装に対し、
**新しい分離構成を優先して読込できるようにする**。

このタスクでは **読込のみ** を実装する。
保存形式の分離はまだ行わない。

---

## このタスクの位置づけ

* タスク1で `config/` ルート導入済み
* `config/config.json` と `config/user/config.json` は手動配置済み前提
* このタスクでは **新しい分離JSONが存在すればそれを読む**
* 無ければ **従来の単一JSON読込へフォールバック** する

---

## 絶対条件

* フック処理を変更しない
* UIレイアウトを変更しない
* 既存のランタイム構造は維持する
* 最小差分で実装する
* JSON保存処理はまだ分離しない
* 旧 `config/user/config.json` 読込互換は維持する

---

## 設計の前提

### 1. 起動入口

起動時は `config/config.json` を読む。
このファイルは **起動入口 + UI共通設定** を持つ。

例:

```json
{
  "keymap_set_path": "user/keymap_sets/default.json",
  "ui_font_delta_pt": 0,
  "last_used_directory": ""
}
```

---

### 2. 分離構成の中心

`keymap_set.json` は **構成セットの設定本体** とする。

役割:

* `trigger_set_path`
* `hotkey_presets_path`
* `active_keymap_path`
* `keymaps[]`
* `hook_stop_key`
* `hook_toggle_key`
* `keyboard_layout`
* `keyboard_show_physical_key_labels`
* `debug_jis_special_key_events`
* `external_keyboard_layouts`

---

### 3. 参照方式

* すべて **configルート基準の相対パス**
* IDは使わない
* `keymap_set.json` の `keymaps` は **方式2**
* `switch_key` は keymap エントリ内に持つ
* 読込時に既存の `keymap_switch_keys` へ合成する

---

### 4. ランタイム構造

読込後の `self.data` は従来のままにする。
つまり分離JSONを読んでも、最終的には既存の `ensure_config_compatibility()` が扱える形へ合成する。

---

## 新しい読込の優先順

以下の優先順で実装すること。

### 優先1

`config/config.json` に `keymap_set_path` があり、参照先の `keymap_set.json` が存在する場合
→ **分離構成として読込**

### 優先2

上記が成立しない場合
→ **従来の単一JSON（`config/user/config.json` または旧フォールバック先）を読込**

---

## 欠損時の扱い

安全側で継続すること。

### ケース1

`config/config.json` が無い、または `keymap_set_path` が無い
→ 単一JSON読込へフォールバック
→ それも無ければ空データ起動

### ケース2

`keymap_set.json` が無い
→ 単一JSON読込へフォールバック
→ それも無ければ空データ起動

### ケース3

`trigger_set_path` が無い、または読めない
→ trigger は空配列で継続
→ messagebox 通知は可

### ケース4

`hotkey_presets_path` が無い、または読めない
→ preset は空配列で継続
→ messagebox 通知は可

### ケース5

`keymaps[]` の一部が無い、または読めない
→ その keymap だけ無効
→ 他は継続
→ `active_keymap_path` が欠損していれば active は未設定または先頭有効keymapへフォールバックしてよい

---

## 実装方針

### 1. 分離読込ロジックは `ConfigService` を境界にする

今回の読込分離の主処理は **ConfigService 側へ寄せる**。
ただし App 側の既存構造を大きく壊さないこと。

想定する責務:

* `config/config.json` の読込
* `keymap_set_path` の解決
* 各参照先JSONの読込
* ランタイム構造への合成
* 不足時フォールバック

---

### 2. パス解決は config ルート基準に統一

`keymap_set_path` や `trigger_set_path` などは、
**configルート基準の相対パス**として解決する helper を用意する。

例:

* `config_root` + `user/keymap_sets/default.json`

親ファイル基準にはしない。

---

### 3. 合成結果は既存互換形にする

最終的に以下を含む辞書にすること。

最低限必要な項目:

* `triggers`
* `hotkey_presets`
* `hook_stop_key`
* `hook_toggle_key`
* `keyboard_layout`
* `keyboard_show_physical_key_labels`
* `debug_jis_special_key_events`
* `external_keyboard_layouts`
* `keymaps`
* `active_keymap_path` 相当から既存アプリが扱える active 情報
* `keymap_switch_keys`

※ 既存コードが `active_keymap_id` を使っているなら、その互換も壊さないこと
※ ただし今回は新保存形式へ合わせた保存はまだ実装しない

---

### 4. keymap 合成ルール

`keymap_set.json` の `keymaps[]` を走査し、読めたものだけランタイムへ積む。

保存形式:

```json
{
  "path": "user/keymaps/default.json",
  "switch_key": "f6"
}
```

読込後のランタイムでは:

* keymap 実体を `data["keymaps"]` に積む
* `switch_key` は `data["keymap_switch_keys"]` に変換する

必要なら keymap 実体側に path 情報を持たせてもよいが、既存互換を壊さないこと。

---

### 5. trigger / preset 合成ルール

* `trigger_set.json` の `triggers` を `data["triggers"]` へ
* `hotkey_presets.json` の `hotkey_presets` を `data["hotkey_presets"]` へ

欠損時は空配列。

---

### 6. 既存単一JSONフォールバック

分離構成が読めない場合は、従来通り `config/user/config.json` を読む。
この処理は残すこと。

---

## 実装候補 helper

名称は既存命名に合わせてよいが、以下のような責務を持つ helper を追加してよい。

* `load_runtime_data(...)`
* `_load_split_config(...)`
* `_load_keymap_set(...)`
* `_load_keymap_entry(...)`
* `_resolve_config_relative_path(...)`
* `_build_runtime_data_from_split(...)`

---

## App 側の変更方針

App 側では、起動時読込の入口を大きく変えず、
**ConfigService に「分離を含めた読込」を委譲する**形を優先する。

つまり、App 側で分離JSONを直接つなぎこまないこと。
分離読込の判断・合成は ConfigService 側に寄せる。

---

## 変更対象の目安

* `keyseq/application/config_service.py`
* 必要なら `keyseq/presentation/app.py`
* 必要なら `keyseq/infrastructure/json_repository.py`

できるだけ読込責務を `ConfigService` に寄せる。
`domain/config.py` の既存互換は壊さない。

---

## 動作確認項目

### 分離構成あり

1. `config/config.json` に `keymap_set_path` があり、参照先一式が存在する
   → 分離構成から起動できる

2. `keymap_set.json` の `trigger_set_path` が無い
   → trigger 空で起動継続

3. `keymaps[]` のうち1件だけ欠損
   → その keymap だけ無効、他は読める

4. `switch_key` がある keymap がランタイムの `keymap_switch_keys` に反映される

---

### フォールバック

5. `config/config.json` に `keymap_set_path` が無い
   → 単一JSON読込へフォールバック

6. `keymap_set.json` が存在しない
   → 単一JSON読込へフォールバック

7. 分離構成も単一JSONも無い
   → 空データ起動

---

### 既存機能

8. 既存UIが壊れていない
9. フック動作が壊れていない
10. direct switch / keymap 管理 / trigger 一覧表示に致命的影響がない

---

## このタスクでやらないこと

* 分離保存
* `config/user/config.json` の廃止処理
* 個別保存
* UIからの keymap_set 切替
* JSON構造の全面移行処理
* ファイルコピーや自動移行機能の追加

---

## 補足判断

* `trigger_set_path` / `hotkey_presets_path` は **今は単数名で開始**
* 将来複数化できる余地は残してよいが、このタスクでは単数前提でよい
* `config.json` のアプリ共通設定は必要最小限に留める
* 既存互換を壊しそうな再設計はしない
