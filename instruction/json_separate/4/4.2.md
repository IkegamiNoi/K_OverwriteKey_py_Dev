# タスク4.2 指示書

**題名:** keymap / trigger_set / sequence の個別保存・個別読込（UI含む）

---

## 🎯 目的

タスク4.1で整理した split 本流構成を拡張し、

* キーマップ
* トリガー一覧
* 出力シーケンス

を個別に保存・読込できるようにする。

この目的は、各要素を個別ファイルとして再利用できる状態にすることである。

---

## ❗ 絶対条件

* フック処理を変更しない
* suppress / 停止制御の仕様を壊さない
* UIスレッドをブロックしない
* 既存JSONキーの削除・意味変更をしない
* 旧形式の読込互換を維持する
* 既存の keymap_set 一括保存・読込を壊さない
* 目的外の再設計をしない

---

# 🧠 設計確定事項

## 1. フォルダ構成

既存継続:

```text
config/user/keymap_sets/
config/user/keymaps/
config/user/trigger_sets/
```

新設:

```text
config/user/sequences/
```

---

## 2. keymap_set の位置づけ

`config/user/keymap_sets/` 配下のファイルは、今まで通り構成セットの入口とする。

この指示書では以後、これを **keymap_set ファイル** と呼ぶ。

keymap_set ファイルは、以下を参照する。

* keymap ファイル
* trigger_set ファイル
* hotkey_presets ファイル
* その他、既存の構成セット設定

---

## 3. trigger_set の保存内容

`config/user/trigger_sets/` 配下には、トリガー一覧を保存する。

trigger_set は以下を持つ。

* トリガーキー
* suppress
* sequence ファイルへの参照

例:

```json
{
  "triggers": [
    {
      "key": "f1",
      "suppress": true,
      "sequence_path": "user/sequences/sample.json"
    }
  ]
}
```

---

## 4. sequence の保存内容

`config/user/sequences/` 配下には、出力シーケンスを保存する。

sequence は以下を持つ。

* label
* run_to_end
* run_to_end_delay_ms
* actions

`run_to_end` / `run_to_end_delay_ms` は UI 上の「連続実行」「間隔(ms)」である。

例:

```json
{
  "label": "sample",
  "run_to_end": false,
  "run_to_end_delay_ms": 300,
  "actions": [
    {
      "type": "hotkey",
      "value": "ctrl+c",
      "label": ""
    }
  ]
}
```

---

## 5. 旧形式互換

既存形式では `triggers[]` 内に以下が入っている。

* label
* run_to_end
* run_to_end_delay_ms
* actions

読込時は旧形式も受け入れる。

### 読込時

* 新形式: `sequence_path` があれば sequence ファイルを読む
* 旧形式: `triggers[]` 内の `label` / `run_to_end` / `run_to_end_delay_ms` / `actions` を読む

### 保存時

* 新形式へ寄せる
* trigger_set には sequence 参照を保存する
* sequence 本体は `config/user/sequences/` または対応する保存先に保存する

---

## 6. パス保存ルール

タスク4.1のルールを継続する。

### 保存時

* config_root 配下 → config_root 基準の相対パス
* config_root 外 → 絶対パス

### 読込時

* 絶対パス → そのまま使用
* 相対パス → config_root 基準で解決

---

## 7. keymap_set がデフォルト外にある場合

keymap_set ファイルをデフォルト以外に保存した場合、
keymap / trigger_set / sequence の個別保存先について確認ダイアログを出す。

確認内容:

* デフォルト保存先へ保存する
* keymap_set ファイル周辺にフォルダを作成して保存する

keymap_set ファイル周辺に保存する場合は、
デフォルト保存先のファイル直上フォルダ名を keymap_set 周辺に作成する。

例:

```text
somewhere/my_set.json
somewhere/keymaps/
somewhere/trigger_sets/
somewhere/sequences/
```

# 💾 保存動作

## 1. 通常保存

通常保存は、現在紐づいている source path へ上書き保存する。

ただし、読込由来フラグが立っている場合は別名保存するか確認する。

---

## 2. 別名で保存

別名で保存ではユーザーに保存先を選ばせる。

保存後は source path を新しい保存先へ更新する。

読込由来フラグは倒す。

---

## 3. 新規作成したものの保存

新規作成した keymap / trigger_set / sequence は、ラベル名を元にファイル名を作成する。

sequence のラベルが日本語の場合も、ラベルをそのままファイル名候補に使う。
ファイル名では Windows 禁止文字のみ `_` に置換する。

trigger_set はラベルを持たないため、現在の keymap_set ファイル名を元にファイル名を作成する。

ラベルが空の場合は以下のような fallback 名を使う。

```text
keymap.json
trigger_set.json
sequence.json
```

同名ファイルが既にある場合は、上書きするか別名にするか確認ダイアログを出す。

---

## 4. 名称変更とラベル連動

別名保存または名称変更時に、
ラベル名の変更を連動させるためのチェックボックスを用意する。

* デフォルトはチェックなし
* チェックなしの場合、ファイル名変更と label は連動しない
* チェックありの場合、保存名に合わせて label も変更する

通常のラベル編集だけでは、ファイル名を自動変更しない。

---

## 5. 読込由来フラグ

keymap / trigger_set / sequence を読込で持ってきた場合、
ランタイム内部で読込由来フラグを管理する。

例:

```text
_keymap_imported
_trigger_set_imported
_sequence_imported
```

保存JSONへ内部フラグを出力しない。

読込由来フラグがあるものを変更して保存する場合は、
別名で保存するか確認する。

別名で保存した場合、読込由来フラグは倒す。

---

# 🧩 ランタイム内部情報

## 1. source path

個別保存・読込のため、ランタイム内部で source path を保持してよい。

例:

```text
_keymap_source_path
_trigger_set_source_path
_sequence_source_path
```

これらは内部管理用であり、単一JSON Export や実体JSON保存時には出力しない。

---

## 2. dirty 管理

未保存管理は個別単位で扱う。

最低限、以下の単位を区別する。

* keymap_set
* keymap
* trigger_set
* sequence

変更分類:

| 操作 | dirty 対象 |
| --- | --- |
| トリガーキー変更 | trigger_set |
| suppress 変更 | trigger_set |
| sequence label 変更 | sequence |
| actions 追加・編集・削除・並び替え | sequence |
| run_to_end 変更 | sequence |
| run_to_end_delay_ms 変更 | sequence |
| keymap mappings 変更 | keymap |
| keymap 切替キー変更 | keymap_set または keymap_set 側参照 |

既存の全体 dirty 表示は維持してよい。
ただし、終了・読込・新規作成などの未保存確認では、個別 dirty を確認対象に含める。

---

# 🛠 実装方針

## 1. ConfigService

保存・読込の実処理は ConfigService 側に寄せる。

追加または整理する処理:

* keymap 個別保存
* keymap 個別読込
* trigger_set 個別保存
* trigger_set 個別読込
* sequence 個別保存
* sequence 個別読込
* trigger_set と sequence の相互変換
* 内部キー除去

App 側で JSON 分解処理を直接持たない。

---

## 2. App

App 側は以下を担当する。

* UIボタン
* ファイルダイアログ
* 上書き / 別名保存確認
* ラベル連動チェックボックス付きダイアログ
* source path / dirty / imported フラグの反映
* UI更新

---

## 3. 既存一括保存との整合

既存の `save_keymap_set` / `save_as` / `load_keymap_set_from` は壊さない。

一括保存では、新しい sequence 分離形式も含めて再読込可能な状態にする。
