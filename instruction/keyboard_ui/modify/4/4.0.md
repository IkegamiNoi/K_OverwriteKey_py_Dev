# タスク4 指示書

## JIS特殊キー対応（`scan_code` 併用の導入）

### 目的

JIS配列で名前解決しづらい特殊キーを、キー名だけでなく `scan_code` でも扱えるようにする。
対象はまず、読み取り失敗が起きている特殊キーに限定する。
全面的な scan_code 移行ではなく、**最小差分の併用対応**にする。

---

## 対象キー

まずは以下を対象にする。

* 無変換
* 変換
* かな
* 半角/全角

必要なら将来的に追加できる構造にしてほしい。

---

## 今回やってほしいこと

### 1. キー識別に `scan_code` を併用できるようにする

現在は主にキー名ベースで扱っているが、
JIS特殊キーについては `event.name` / `keysym` だけでは不安定なので、`scan_code` も使えるようにしてほしい。

方針:

* 通常キーは従来どおりキー名ベースでよい
* 特殊キーのみ、必要に応じて `scan_code` を優先または補助利用する

---

### 2. レイアウト定義に `scan_code` を持てるようにする

外部レイアウトJSONおよび内部レイアウト定義で、各キーに任意で `scan_code` を持てるようにしてほしい。

必須ではなく任意項目でよい。

例:

```json
{
  "id": "muhenkan",
  "label": "無変換",
  "x": 3.0,
  "y": 5.2,
  "w": 1.5,
  "h": 1.0,
  "scan_code": 123
}
```

今回は scan_code 値の完全網羅までは不要。
まずは構造を受け入れられるようにする。

---

### 3. キーボードUI編集時のキー取得を改善

キーボードUIでキーをクリックして置換先を入力する処理について、
JIS特殊キーだけは `keysym` だけでなく scan_code ベースでも取得できるようにしてほしい。

方針:

* 通常キーは今のままでよい
* 特殊キーだけ補助マッピングを入れる
* 既存の Esc キャンセル / 右クリッククリアは壊さない

---

### 4. 実行時の表示・マッチングで特殊キーを扱えるようにする

以下のどちらか、または両方で使えるようにしてほしい。

* 名前ベースで解決できる場合は従来どおり
* 名前で無理な場合は scan_code ベースで補う

目的は、JIS特殊キーが

* レイアウト表示で正しく対応づけられる
* キーマップ編集で入力できる
* 実行時にキーとして認識される

ようにすること。

---

## 実装制約

* 全キーを scan_code ベースへ全面移行しない
* 既存の通常キー処理はなるべく変えない
* 既存の外部レイアウトJSONを壊さない
* `scan_code` は追加の任意情報として扱う
* フックの方式そのものは変更しない
* suppress 挙動を壊さない

---

## バリデーション追記

レイアウトJSONの各キーに `scan_code` がある場合は以下を許可する。

* int
* 数値文字列から int 変換可能なもの

不正ならエラーにする。
`scan_code` が無い場合は従来どおり有効。

---

## 実装の方向性

以下のような最小差分で進めてほしい。

1. `KeySpec` に任意の `scan_code` を追加
2. JSON読込・バリデーションで `scan_code` を受け付ける
3. 特殊キー向けの `scan_code -> 内部キー名` 補助マップを追加
4. UI編集時に特殊キー取得で補助利用
5. 必要なら KeyboardWindow の表示対応づけにも補助利用

---

## 禁止事項

* 全面的な内部ID再設計はしない
* 記号キーIDの大整理はこのタスクに混ぜない
* keymap切替UIの追加はしない
* keymap_sets / trigger_sets 分離はしない

---

## 出力形式

* どの箇所で scan_code を併用したか簡潔に説明
* 動作確認項目も提示すること

---

## 動作確認項目

1. 無変換キーをキーボードUI編集で取得できる
2. 変換キーをキーボードUI編集で取得できる
3. かなキーをキーボードUI編集で取得できる
4. 半角/全角キーをキーボードUI編集で取得できる
5. これらのキーを含むレイアウトJSONを読み込んでもエラーにならない
6. 通常キーの編集・表示・実行が壊れていない
7. 既存の US レイアウト表示が壊れていない
8. 既存の trigger / keymap / 予約キー表示が壊れていない

---

## 完了条件

* JIS特殊キーを名前だけでなく scan_code でも補助的に扱える
* 通常キー処理は壊れていない
* 外部レイアウトJSON互換は維持されている
