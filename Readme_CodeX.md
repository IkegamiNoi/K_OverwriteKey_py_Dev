# プロジェクト概要
Python + tkinter + keyboard ライブラリを使用したキーボード入力置換シーケンサーアプリ。

主な機能
- 複数トリガー対応
- 出力シーケンス（hotkey / text / mouse_click）
- OSショートカットプリセット機能
- フック停止トリガー機能
- JSON保存 / 読み込み対応
- 連続実行機能（run_to_end）

---

# 絶対に守る設計方針

## 1. 構成
- 単一ファイル構成を維持する →　オニオンアーキテクチャへと変更する
- tkinter をGUIとして使用する
- keyboard ライブラリを使用する
- グローバルフック方式を維持する

グローバルフック仕様
- トリガー検知：keyboardで行う（アプリ外でも反応させるため）
- 記録（hotkey入力のキャプチャ）：OSショートカットはプリセット/手入力（危険なため）

---

## 2. トリガー仕様

通常トリガー
- 単キーのみ

停止トリガー
- 単キーのみ
- キャプチャ入力のみ
- suppress=True

制約
- 停止トリガーと通常トリガーの重複は禁止

---

## 3. フック管理仕様

- suppress=True を使用
- 編集系ダイアログ表示中はフックを一時停止
- 一時停止はネスト対応（カウンタ方式）
- フック暴走を起こす設計は避ける
- UIスレッドをブロックしない

---

## 4. JSON構造（変更禁止・互換維持）

基本構造

{
  "triggers": [
    {
      "key": "f1",
      "label": "",
      "suppress": true,
      "actions": [
        {
          "type": "hotkey",
          "value": "ctrl+c",
          "label": ""
        }
      ]
    }
  ],
  "hotkey_presets": [
    {"label": "Win+D", "value": "windows+d"}
  ],
  "hook_stop_key": "f12"
}

ルール
- 既存キーを削除しない
- 新キー追加時は後方互換を必ず保つ
- 未定義キーは無視して読み込む

---

## 5. JSON拡張キー（後方互換）

トリガーごとに追加される可能性があるキー

run_to_end
- 型：bool
- 連続実行の有効化

run_to_end_delay_ms
- 型：int
- 連続実行の実行間隔(ms)
- デフォルト：300

例

{
  "key": "f1",
  "label": "",
  "suppress": true,
  "run_to_end": true,
  "run_to_end_delay_ms": 300,
  "actions": [...]
}

---

## 6. 表示形式（統一ルール）

トリガー一覧
01. key: label

出力シーケンス
01. [type] value: label

プリセット一覧
01. value: label

ルール
- ラベルは空でも可
- ラベル空の場合は「:」を表示しない

hotkey

- 不正値は messagebox で通知
- アプリは停止しない

OSショートカット
- プリセット方式を推奨

マウスクリック

(x, y, button, clicks)

例

(500, 300, left, 1)

UIは2画面構成

FullView
- すべての編集機能

CompactView

- プリセット

## 9. 連続実行機能（run_to_end）

トリガーごとに設定

UI
- 出力シーケンスの操作列に
- 「連続実行」チェックボックス
- 「実行間隔(ms)」入力欄

動作

トリガー押下
→ 現在のシーケンス位置から最後まで実行

完走
→ 連続実行停止
→ 次回は先頭から

連続実行中
- 同トリガー押下 → 一時停止
- 再押下 → 再開

制約
- 実行中は他トリガーを抑止
- フック停止トリガーは除外

---

## 10. UI同期ルール

トリガー選択変更時

同期対象
- suppress
- run_to_end
- run_to_end_delay_ms

同期関数

_sync_run_to_end_ui()

---

## 11. UIスレッドルール

GUI更新は必ずUIスレッドで行う

keyboardフックは別スレッドで呼ばれる可能性がある

そのため

self.after()

を使用してUI更新する

---

## 12. コード提案ルール

- 既存レイアウト（grid行）を不用意に変更しない
- 既存機能を壊さない最小差分を優先する
- 既存の命名・設計を尊重する
- 再設計を勝手に行わない

---

## 13. 安全設計優先

- フックでUIを閉じ込めない
- Alt+Tab等の危険ショートカットに配慮
- suppressの副作用を考慮する
- keyboard例外はtry/exceptで吸収する

---

## 14. パフォーマンス方針

- 無駄な再フックを避ける
- フック登録/解除は必要最小限
- UIスレッドで重い処理を行わない
